# 回滚说明

## 1. 回滚目标

本方案从设计上要求“随时可撤销”。

回滚的含义是：

- 停止让 live workspace 通过 symlink 指向 repo skill
- 恢复迁移前的目录或文件状态
- 保持运行态文件不受影响
- 全程**不碰 Gateway**

---

## 2. 可回滚的前提

因为本方案采用以下约束，所以天然支持回滚：

- **skill 用逐目录 symlink**，撤销粒度小
- **模板 copy_if_missing**，默认不覆盖原文件
- **运行态文件不入 repo**，无需从 repo 恢复运行数据
- **不碰 Gateway**，不依赖服务重启恢复

---

## 3. 标准回滚场景

### 场景 A：某个 skill 的 symlink 需要撤回

处理思路：

1. 删除该 skill 的 live symlink
2. 恢复原始本地目录，或改回上一版目录
3. 验证 skill 读取路径恢复正常

因为是逐目录 symlink，所以只影响单个 skill，不波及其他目录。

### 场景 B：模板初始化后发现不想使用

由于模板采用 copy_if_missing，通常只有“新生成的文件”需要处理。

处理思路：

- 如果该模板文件是本次新建、且确认未被人工修改，可删除
- 如果该文件后来已经被人工编辑，应视为本地文件，谨慎处理，不应被 repo 强制回收

### 场景 C：迁移计划整体暂停

处理思路：

- 停止继续迁移后续 skill
- 对已迁移 skill 逐个撤销 symlink
- 保留 repo 文档，供后续重新评估

---

## 4. 不需要回滚的内容

以下内容本来就不应该被 repo 接管，因此不存在“从 repo 回滚”的问题：

- `memory/`
- `reports/`
- `.learnings/`
- 本地日志/缓存
- Gateway 运行态

一句话：**运行态文件不入 repo。**

---

## 5. 回滚检查清单

执行回滚后，检查：

- 对应 skill 的 symlink 是否已移除
- live workspace 是否恢复到预期目录状态
- 其他 skill 是否未受影响
- 模板文件是否仅按需要保留/移除
- `memory/`、`reports/`、`.learnings/` 是否保持原状
- 是否完全没有 Gateway 相关变更

---

## 6. 回滚策略建议

### 小步迁移，小步回滚

建议始终：

- 一次只迁移 1 个 skill 或一小组 skill
- 每迁移一步就验证一步
- 出现问题立即按 skill 粒度回滚

这样做的好处：

- 问题范围小
- 原因容易定位
- 回滚成本低

### 模板优先保守

对于模板文件：

- 宁可少复制，也不要误覆盖
- 一律遵守 copy_if_missing
- 发现边界不清时，不自动处理

---

## 7. 一句话回滚原则

> skill 用逐目录 symlink，模板 copy_if_missing，运行态文件不入 repo，不碰 Gateway，可回滚。
