# Research modes

Pick the mode from downstream use, not just the user’s wording.

## 1. exploration

Use when the user is mapping a space and wants breadth before judgment.

### Goal
- build a landscape map
- identify major concepts, players, fault lines, and open questions
- show where deeper follow-up is most valuable

### Default behavior
- favor breadth over exhaustive detail
- avoid premature conclusions
- use fewer lanes unless the space naturally splits into distinct clusters

### Typical outputs
- taxonomy / landscape map
- key players or approaches
- first-pass comparisons
- unresolved questions
- suggested next-step lanes

## 2. decision-support

Use when the real goal is to choose, prioritize, recommend, or compare.

### Goal
- reduce ambiguity for a real decision
- surface tradeoffs and recommendation logic

### Default behavior
- bias toward comparisons and implications
- separate must-know evidence from nice-to-know context
- optimize for actionability, not encyclopedic coverage

### Typical outputs
- recommendation
- comparison matrix
- tradeoffs / risks
- evidence table
- suggested action path

## 3. executive-brief

Use when the output feeds a leader, memo, meeting, or presentation.

### Goal
- compress complexity into a concise, conclusion-first brief

### Default behavior
- lead with judgments
- keep structure crisp
- compress hard without losing key uncertainty

### Typical outputs
- 3-7 key judgments
- one-page summary
- supporting evidence bullets
- major uncertainties
- recommended next actions

## 4. technical-due-diligence

Use when the user needs a serious technical assessment of a model, product, system, framework, or architecture.

### Goal
- determine what is technically real, differentiated, scalable, or risky
- separate demos and claims from underlying substance

### Default behavior
- prioritize Tier 1 sources
- scrutinize benchmark quality and missing baselines
- apply stronger red-team pressure than other modes

### Typical outputs
- architecture breakdown
- benchmark / eval assessment
- implementation or deployment considerations
- technical risks and unknowns
- strategic technical judgment

## Selection hints

- “先摸清楚/梳理全景/建立地图” -> exploration
- “帮我比较/判断/选型/给建议” -> decision-support
- “给老板看/汇报/一页纸/高层摘要” -> executive-brief
- “技术上到底怎么样/benchmark 靠不靠谱/架构是否成立” -> technical-due-diligence

If the task needs both landscape and recommendation, keep exploration in framing but produce a decision-support output.
